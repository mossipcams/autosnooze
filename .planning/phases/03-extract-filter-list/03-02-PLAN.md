**CRITICAL: Event handling workaround (established pattern from Phase 2):**

Lit's `@event` template bindings do NOT fire in jsdom when events are dispatched from child custom elements across shadow DOM boundaries. This was established in Phase 2 (02-02-SUMMARY.md).

For selection change events, tests that previously did:
```javascript
card._toggleSelection('automation.test');
// then checked card._selected
```

Must now do one of:
- **Option A (preferred):** Call the child method AND manually invoke the parent handler:
  ```javascript
  const al = queryAutomationList(card);
  al._toggleSelection('automation.test');
  // Since jsdom doesn't propagate the composed event to Lit template bindings,
  // manually invoke the parent handler:
  card._handleSelectionChange(new CustomEvent('selection-change', {
    detail: { selected: ['automation.test'] },
  }));
  await card.updateComplete;
  ```

- **Option B (simpler for tests that just need state):** Directly set `card._selected`:
  ```javascript
  card._selected = ['automation.test'];
  await card.updateComplete;
  ```
  This works because `_selected` is still a `@state()` on the parent. Use this when the test is verifying parent behavior (like the snooze button) rather than the selection UI.

**Note on child updateComplete:** After setting properties on the child component, await BOTH updates:
```javascript
const al = queryAutomationList(card);
al._filterTab = 'areas';
await al.updateComplete;
await card.updateComplete; // parent may also need to re-render
```
